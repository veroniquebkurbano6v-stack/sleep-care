# 睡眠评估干预系统 - 项目规则

本规则定义睡眠评估干预系统的开发规范，AI助手在所有项目对话中必须严格遵守。

---

## 1. 技术栈规范

### 1.1 前端技术栈
- **框架**：微信小程序原生框架（WXML/WXSS/JavaScript）
- **UI组件**：使用微信原生组件，不引入第三方UI框架
- **状态管理**：小程序页面级状态管理，复杂场景可使用简单的全局状态
- **图表库**：使用ECharts微信小程序版或微信原生Canvas API

### 1.2 后端技术栈
- **语言**：Node.js 18+
- **框架**：Express.js 或 Koa.js
- **数据库**：MySQL 8.0+
- **ORM**：Sequelize 或 Prisma
- **认证**：JWT (JSON Web Token)
- **消息队列**：Redis (可选，用于异步任务)

### 1.3 部署规范
- **云服务**：腾讯云或阿里云
- **容器化**：Docker (可选)
- **CI/CD**：GitHub Actions 或 GitLab CI

## 2. 数据库规范

### 2.1 命名规范
- **表名**：使用小写英文单词，单词之间用下划线分隔（snake_case）
- **字段名**：使用小写英文单词，单词之间用下划线分隔（snake_case）
- **索引名**：前缀为 `idx_` + 表名 + 字段名
- **外键名**：前缀为 `fk_` + 表名 + 字段名
- **唯一键名**：前缀为 `uk_` + 表名 + 字段名

### 2.2 数据类型规范
- **主键**：使用 `BIGINT` 类型，配合 `AUTO_INCREMENT`
- **手机号**：使用 `VARCHAR(20)`，不使用数字类型
- **时间戳**：使用 `DATETIME` 类型，统一使用 `DEFAULT CURRENT_TIMESTAMP`
- **布尔值**：使用 `TINYINT` (0/1)，不使用 `BOOLEAN`
- **枚举值**：使用 `TINYINT` 配合常量定义，不使用 `ENUM`
- **JSON数据**：使用 `TEXT` 类型存储，便于扩展

### 2.3 约束规范
- **非空约束**：必填字段必须设置 `NOT NULL`
- **默认值**：可选字段应设置合理的默认值
- **外键约束**：关联表必须设置外键约束，定义 `ON DELETE` 行为
- **唯一约束**：需要唯一的字段必须设置唯一约束
- **索引**：查询频繁的字段必须建立索引，避免全表扫描

### 2.4 数据安全规范
- **敏感数据**：手机号、身份证号等敏感信息需加密存储或脱敏处理
- **数据备份**：定期进行全量备份和增量备份
- **权限控制**：通过角色和授权表控制数据访问权限

## 3. API规范

### 3.1 接口风格
- **架构风格**：RESTful API
- **协议**：HTTPS
- **端口**：默认使用443端口

### 3.2 URL规范
- **基础路径**：`/api/v1`
- **资源路径**：使用复数名词，如 `/api/v1/users`
- **版本控制**：通过URL路径进行版本控制，如 `/api/v1`, `/api/v2`

### 3.3 HTTP方法规范
- **GET**：查询资源，幂等操作
- **POST**：创建资源
- **PUT**：更新资源（完整替换）
- **PATCH**：更新资源（部分更新）
- **DELETE**：删除资源

### 3.4 请求规范
- **Content-Type**：`application/json`
- **认证方式**：在请求头中携带 `Authorization: Bearer <token>`
- **参数校验**：所有请求参数必须进行严格校验

### 3.5 响应规范

#### 成功响应
```json
{
    "code": 200,
    "message": "success",
    "data": {}
}
```

#### 错误响应
```json
{
    "code": 400,
    "message": "error message",
    "data": null
}
```

#### 响应码规范
| 响应码 | 含义 |
|--------|------|
| 200 | 请求成功 |
| 400 | 请求参数错误 |
| 401 | 未认证或认证失败 |
| 403 | 无权限访问 |
| 404 | 资源不存在 |
| 500 | 服务器内部错误 |

### 3.6 分页规范
```json
{
    "code": 200,
    "message": "success",
    "data": {
        "list": [],
        "total": 100,
        "page": 1,
        "pageSize": 10
    }
}
```

## 4. 代码规范

### 4.1 文件命名规范
- **组件文件**：使用小写英文，单词之间用连字符分隔，如 `sleep-report.wxml`
- **页面文件**：使用小写英文，单词之间用连字符分隔，如 `sleep-report.js`
- **工具函数**：使用小写英文，单词之间用下划线分隔，如 `date_utils.js`
- **常量文件**：使用全大写英文，单词之间用下划线分隔，如 `CONSTANTS.js`

### 4.2 代码风格规范
- **缩进**：使用2个空格进行缩进
- **分号**：每行代码结束必须使用分号
- **空格**：运算符两侧、括号内侧必须有空格
- **逗号**：数组和对象最后一个元素后必须加逗号
- **引号**：统一使用单引号

### 4.3 变量命名规范
- **普通变量**：使用驼峰命名法（camelCase），如 `sleepScore`
- **常量**：使用全大写，单词之间用下划线分隔，如 `MAX_SLEEP_SCORE`
- **类名**：使用大驼峰命名法（PascalCase），如 `SleepReport`
- **函数名**：使用驼峰命名法，动词开头，如 `getSleepReport`

### 4.4 函数规范
- **函数长度**：单个函数代码不超过50行
- **函数参数**：参数数量不超过4个，过多参数使用对象传递
- **函数注释**：每个函数必须有JSDoc风格的注释
- **错误处理**：必须进行错误捕获和处理，避免全局异常

### 4.5 代码质量规范
- **代码复用**：重复代码必须抽取为公共函数或组件
- **性能优化**：避免不必要的计算和重复渲染
- **内存管理**：及时释放不再使用的资源
- **代码审查**：所有代码提交前必须经过代码审查

## 5. 注释规范

### 5.1 注释原则
- **必要性**：代码本身能表达清楚的逻辑不需要注释
- **清晰性**：注释必须清晰、准确，避免模糊表述
- **更新性**：代码修改时必须同步更新相关注释
- **完整性**：复杂逻辑和业务规则必须有详细注释

### 5.2 文件头部注释
每个文件必须包含文件头部注释，说明文件用途、作者和创建时间：

```javascript
/**
 * 睡眠报告服务模块
 * 负责睡眠报告的CRUD操作
 * @author Developer
 * @created 2024-01-01
 */
```

### 5.3 函数注释
每个函数必须包含JSDoc风格的注释：

```javascript
/**
 * 获取用户睡眠报告列表
 * @param {number} userId - 用户ID
 * @param {object} options - 查询选项
 * @param {string} options.startDate - 开始日期
 * @param {string} options.endDate - 结束日期
 * @returns {Promise<Array>} 睡眠报告列表
 */
async function getSleepReports(userId, options) {
    // 函数实现
}
```

### 5.4 复杂逻辑注释
复杂业务逻辑和算法必须有详细的注释说明：

```javascript
// 根据睡眠阶段序列计算深睡占比
// stage_curve: 每30秒一个值的数组，0=清醒,1=浅睡,2=深睡,3=REM
// 深睡占比 = 深睡时长 / 总睡眠时长 * 100
const deepCount = stageCurve.filter(stage => stage === 2).length;
const sleepCount = stageCurve.filter(stage => stage !== 0).length;
const deepRatio = sleepCount > 0 ? (deepCount / sleepCount * 100).toFixed(1) : 0;
```

### 5.5 注释格式规范
- **单行注释**：使用 `//`，注释内容前有一个空格
- **多行注释**：使用 `/* */`，适用于较长的注释
- **JSDoc注释**：使用 `/** */`，适用于函数和类的文档注释

## 6. AI行为规范

### 6.1 AI使用原则
- **辅助开发**：AI作为辅助工具，不能替代开发者的思考和决策
- **代码审查**：AI生成的代码必须经过人工审查和测试
- **知识验证**：AI提供的技术方案和建议必须进行验证
- **数据安全**：不得向AI泄露敏感数据和机密信息

### 6.2 AI提示词规范
- **明确性**：提示词必须明确、具体，避免模糊表述
- **完整性**：提供足够的上下文信息，帮助AI理解需求
- **结构化**：使用清晰的结构和格式组织提示词内容
- **版本控制**：重要的提示词应进行版本管理

### 6.3 AI生成代码审查要点
- **安全性**：检查是否存在安全漏洞，如SQL注入、XSS攻击等
- **正确性**：验证代码逻辑是否符合业务需求
- **规范性**：检查代码是否符合项目的代码规范
- **性能**：评估代码的性能表现，是否存在优化空间

### 6.4 AI使用禁忌
- **禁止行为**：
  - 不得使用AI生成未经验证的生产代码
  - 不得向AI提供数据库连接信息、密钥等敏感数据
  - 不得依赖AI进行架构设计和技术决策
  - 不得使用AI生成虚假测试数据
- **推荐行为**：
  - 使用AI辅助编写单元测试用例
  - 使用AI生成代码注释和文档
  - 使用AI进行代码重构建议
  - 使用AI辅助解决技术难题

### 6.5 AI协作流程
```
需求分析 → AI辅助设计 → 人工审查 → 编码实现 → AI辅助测试 → 人工验证 → 部署上线
```
