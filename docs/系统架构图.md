# 睡眠评估干预系统 - 系统架构图

本系统采用前后端分离架构，主要由以下组件构成，支撑MVP功能的完整运行。

---

## 系统架构组件

### 1. 微信小程序（前端）
- **职责**：作为用户直接交互界面，负责展示数据、接收操作指令
- **技术栈**：微信原生框架
- **安全机制**：JWT认证机制
- **兼容性**：适配iOS/Android双平台

### 2. 云后台API服务（后端）
- **职责**：基于RESTful接口提供用户管理、设备管理、睡眠报告查询、e-CBTi数据处理等核心服务
- **部署方式**：云端服务器
- **接口规范**：RESTful标准

### 3. 数据库（MySQL）
- **职责**：持久化存储用户信息、设备状态、睡眠报告、睡眠日记、量表结果等核心数据
- **特性**：支持高效查询与事务处理

### 4. 模拟数据生成器（后台任务）
- **职责**：每日凌晨2点为绑定虚拟设备的用户生成符合生理规律的模拟睡眠数据
- **用途**：用于无硬件阶段的功能验证

### 5. 医生端Web后台
- **职责**：提供独立认证入口，供医生查看授权患者的脱敏数据
- **功能**：查看报告、日记、测评结果，支持添加干预建议

### 6. 虚拟设备模拟器
- **职责**：替代真实硬件上报心跳、状态，接收配置更新
- **用途**：用于测试设备绑定、在线状态等场景

### 7. OTA固件管理模块
- **职责**：预留接口，支持上传新版本固件、发布更新日志、标记强制升级
- **用途**：为后续硬件接入做准备

### 8. 消息推送服务
- **职责**：基于微信订阅消息机制推送提醒
- **推送场景**：
  - 早晨推送"睡眠报告已生成"提醒
  - 设备离线超24小时时发送告警

## 数据流向

### 数据流1：用户请求数据
```
用户在微信小程序发起请求
    ↓
调用云后台API
    ↓
查询数据库返回数据
    ↓
小程序渲染展示
```

### 数据流2：模拟数据生成
```
模拟数据生成器每日定时运行
    ↓
生成睡眠数据
    ↓
写入数据库sleep_reports表
```

### 数据流3：医生查看患者数据
```
医生登录Web后台
    ↓
验证身份
    ↓
查询doctor_authorizations表
    ↓
获取患者列表
    ↓
调用API获取脱敏报告数据
```

### 数据流4：OTA固件更新
```
OTA管理模块上传固件
    ↓
API暴露最新版本信息
    ↓
小程序检查更新
    ↓
用户确认安装
    ↓
记录日志（硬件接入后触发真实升级）
```

### 数据流5：消息推送
```
消息推送服务监听事件
    ↓
检测睡眠报告生成 / 设备离线
    ↓
调用微信订阅消息API
    ↓
推送通知到用户小程序
```

---

## 架构特点

1. **无硬件依赖开发**：待真实设备就绪后仅需替换数据源与控制指令下发模块即可无缝对接
2. **良好的演进性**：预留OTA、消息推送等扩展接口，支持未来功能迭代
3. **安全性保障**：JWT认证、脱敏数据、微信订阅消息机制多重安全保障
4. **高可用性**：前后端分离，各模块独立部署，便于水平扩展
