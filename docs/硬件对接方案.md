# 睡眠监测硬件对接方案

> 本文档描述将当前「虚拟模拟数据」替换为真实睡眠监测硬件的数据接入方案。

---

## 一、数据链路改造

### 1.1 当前架构 vs 目标架构

```
【当前架构】
小程序 → 后端API → SQLite(sql.js) → 伪随机生成数据

【目标架构】
睡眠硬件(MQTT) → 边缘网关 → 后端API → MySQL → 真实采集数据
```

### 1.2 硬件通信协议选择

| 协议 | 适用场景 | 优势 | 劣势 |
|------|---------|------|------|
| **MQTT** | 低功耗穿戴设备 | 轻量、断线重连、发布订阅模式 | 需要 Broker 中转 |
| Bluetooth LE (BLE) | 近距离直连 | 无需网络、低延迟 | 距离限制(~10m)、需配对 |
| HTTP/WebSocket | Wi-Fi 设备 | 通用性强 | 功耗较高 |

**推荐方案：MQTT（主）+ BLE（备用直连）**

### 1.3 MQTT Topic 设计

```
# 设备上报原始数据
devices/{serial_no}/raw          # 心率/体动/呼吸率原始采样
devices/{serial_no}/noise        # 环境噪音分贝值
devices/{serial_no}/status       # 设备在线状态/电量

# 服务端下发指令
commands/{serial_no}/sync        # 同步时间/固件更新
commands/{serial_no}/config      # 配置采样频率/阈值
commands/{serial_no}/reset       # 重启设备

# 系统事件
events/device/{serial_no}/online   # 设备上线
events/device/{serial_no}/offline  # 设备离线
```

### 1.4 数据流代码改造要点

**新增文件：**
- `backend/services/mqtt-client.js` — MQTT 客户端连接与消息分发
- `backend/services/data-processor.js` — 原始数据处理（滤波、分段、聚合）

**修改文件：**
- `backend/app.js` — 新增 `/api/devices/mqtt/config` 接口返回 MQTT 连接参数
- `backend/app.js` — 睡眠报告生成逻辑从「伪随机」改为「读取已入库的真实数据」

## 二、设备绑定与指令下发

### 2.1 绑定流程（BLE 配对模式）

```
用户操作                      后端                        硬件
  |                           |                           |
  |-- 1.点击"添加设备" ------->|                           |
  |<-- 2.返回配对Token --------|                           |
  |                           |                           |
  |-- 3.BLE扫描附近设备 -------|--                         |
  |<-- 4.发现设备列表 ---------|                           |
  |                           |                           |
  |-- 5.选择设备+输入配对码 --->|                           |
  |                           |-- 6.验证Token ------------>|
  |                           |<-- 7.确认 ----------------|
  |<-- 8.绑定成功 -------------|                           |
```

### 2.2 设备指令接口设计

```yaml
# POST /api/v1/devices/{id}/command
# 下发控制指令到指定设备
body:
  type: string        # sync | config | reset | start_monitor | stop_monitor
  params: object      # 指令参数（如采样频率、报警阈值等）
response:
  command_id: string  # 指令 ID（用于追踪执行结果）
  status: pending     # pending | sent | executed | failed
```

### 2.3 指令状态轮询 / MQTT 回调

```yaml
# GET /api/v1/devices/{id}/command/{commandId}
# 查询指令执行状态
response:
  command_id: string
  type: string
  status: executed | failed
  result: object       # 执行结果详情
  executed_at: datetime
```

## 三、数据库迁移方案（SQLite → MySQL）

### 3.1 迁移步骤

```
步骤 1: 备份现有 SQLite 数据
步骤 2: 在 MySQL 创建目标库和表结构（migration.sql）
步骤 3: 用 migrate-data.js 导出 SQLite 数据 → 导入 MySQL
步骤 4: 修改 .env 设置 DATABASE_TYPE=mysql
步骤 5: 重启后端，验证数据完整性
步骤 6: 对比查询结果，确认无误后切流量
```

### 3.2 DDL 差异对照

| 特性 | SQLite | MySQL |
|------|--------|-------|
| 主键自增 | `INTEGER PRIMARY KEY AUTOINCREMENT` | `BIGINT AUTO_INCREMENT` |
| 字符串类型 | `VARCHAR(N)` | `VARCHAR(N)` (相同) |
| 时间戳 | `DATETIME DEFAULT CURRENT_TIMESTAMP` | `DATETIME DEFAULT CURRENT_TIMESTAMP` (相同) |
| JSON 存储 | `TEXT` (手动序列化) | `JSON` 类型 (原生支持) |
| 外键约束 | 支持 | 支持 |
| 全文搜索 | FTS5 插件 | FULLTEXT 索引 |

### 3.3 迁移脚本使用方法

```bash
# 1. 确保 MySQL 已启动且目标库已建好
mysql -u root -p < docs/migration.sql

# 2. 执行数据迁移（读取 sleep_care.db → 写入 MySQL）
DATABASE_TYPE=mysql MYSQL_HOST=localhost node docs/migrate-data.js

# 3. 验证行数一致性
echo "SQLite:" && sqlite3 sleep_care.db "SELECT COUNT(*) FROM sleep_reports"
echo "MySQL:"  && mysql -u root -e "SELECT COUNT(*) FROM sleep_care.sleep_reports"
```

### 3.4 注意事项

- **user_id 自增值差异**：迁移后 MySQL 的 AUTO_INCREMENT 可能不连续，需在 `migrate-data.js` 中做 ID 映射或重置
- **JSON 字段**：`sleep_stages_json` 和 `noise_json` 从 TEXT 可升级为 MySQL JSON 类型以支持 JSON 函数查询
- **回滚方案**：保留原 SQLite 文件作为备份，随时可切回 `DATABASE_TYPE=sqlite`

## 四、部署与运维

### 4.1 生产环境架构

```
                    ┌─────────────┐
                    │  Nginx 反向代理│
                    │  (HTTPS/443) │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌──────────┐ ┌─────────┐ ┌──────────┐
        │ API 服务  │ │ MQTT   │ │ WebSocket│
        │ (PM2×2)  │ │ Broker  │ │ 推送服务 │
        └────┬─────┘ └────┬────┘ └────┬─────┘
             │            │           │
             └──────┬─────┴───────────┘
                    ▼
              ┌──────────┐
              │  MySQL   │
              │ (主从)   │
              └──────────┘
```

### 4.2 PM2 进程管理配置

```javascript
// ecosystem.config.js
module.exports = {
  apps: [{
    name: 'sleep-care-api',
    script: 'backend/app.js',
    instances: 2,
    exec_mode: 'cluster',
    env: {
      NODE_ENV: 'production',
      PORT: 3000,
      DATABASE_TYPE: 'mysql',
      MYSQL_HOST: '127.0.0.1',
    },
  }],
};
```

### 4.3 监控指标

| 指标 | 采集方式 | 告警阈值 |
|------|---------|---------|
| API 响应时间 | PM2 + 自定义中间件 | P99 > 2000ms |
| 错误率 | 统一错误处理中间件计数 | > 1% |
| MQTT 消息堆积 | Redis LIST 长度 | > 10000 条 |
| MySQL 慢查询 | slow_query_log | > 100ms |
| 在线设备数 | devices 表 last_active_time | 心跳超时 5 分钟 |

### 4.4 日志规范

```
日志格式: [时间] [级别] [模块] 消息
示例:     [2026-06-24T10:30:00Z] [INFO ] [mqtt] Device VIRxxx connected
          [2026-06-24T10:30:01Z] [ERROR] [api] /api/sleep/stages 500 - DB timeout
```

输出目标：
- 开发环境：控制台 stdout
- 生产环境：写入 `/var/log/sleep-care/` 目录，配合 logrotate 轮转

### 4.5 安全加固清单

- [ ] 所有 API 强制 HTTPS（Nginx SSL 终结）
- [ ] JWT Secret 使用强随机字符串（≥32字符）
- [ ] MQTT 启用 TLS + 客户端证书认证
- [ ] MySQL 仅允许内网访问（bind-address=127.0.0.1）
- [ ] 敏感字段加密存储（手机号 AES-256）
- [ ] API 限流（rate-limiting middleware，100 req/min/IP）
- [ ] 定期备份数据库（cron + mysqldump 每日凌晨 3 点）
